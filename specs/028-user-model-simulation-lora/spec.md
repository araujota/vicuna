# Feature Specification: User Model Expansion and Counterfactual User Simulation

**Feature Branch**: `023-adam-runtime-updates`  
**Created**: 2026-03-13  
**Status**: Draft  
**Input**: User description: "with regard to item 6 of your new audit report: evaluate the current state of the user model we maintain in hard memory and through representation in self-state registers, and research and evaluate ways in which we can capture the highest-leverage additional pieces in those two states. another layer we must implement is a separate LoRA representing the rhetorical biases of the user of the system, built cumulatively from only the \"user messages\" in evicted spans. this user personality substrate should not be versioned explicitly like the temporal lora stack is, it should be continuously modified and remain at a fixed size. this lora should only be swapped in during counterfactual simulation in dmn loops as a crude simulation of user response to new messages. when invoking the user personality substrate LoRA, ALL model-specific temporal loras must be ablated such that the simulated user response comes only from the base model and user personality lora. this user-bias model is how we will test, crudely, whether certain counterfactuals would have generated more favorable changes in self-state as a result of user response. in the flow I'm describing, the system would run/plan/generate under counterfactual conditions and generate a message, swap to only user lora + base model, \"send\" the message, receive the response treated as a message from the user, swap back to the normal inference stack, observe the change in self-state generated by the simulated user response, and proceed. research, architect and implement these changes at a production level using all the tools at your disposal"

## User Scenarios & Testing

### User Story 1 - Rich User Model Surfaces (Priority: P1)

The runtime should maintain a materially richer user model than the current
social residue. Hard memory and self-state must preserve the highest-leverage
user-specific information in typed, bounded form so planning and allostatic
control can use it without collapsing into free-form profile text.

**Why this priority**: Without a stronger user model, later simulation and
counterfactual scoring remain grounded in shallow trust or dissatisfaction
signals rather than actual user preferences and rhetorical tendencies.

**Independent Test**: Ingest representative user and tool episodes, then verify
that hard memory archives richer `USER_MODEL` fragments and that self-state
exposes stable preference and rhetorical summary fields through typed APIs.

**Acceptance Scenarios**:

1. **Given** repeated user exchanges with stable formatting and autonomy cues,
   **When** the self-state processes those events, **Then** the model state
   exposes bounded estimates for user format preference, directness,
   clarification propensity, and autonomy preference.
2. **Given** a socially significant or stylistically informative user exchange,
   **When** hard memory archives postwrite primitives, **Then** the archival
   batch includes a richer `USER_MODEL` primitive that captures preference and
   rhetorical residue rather than only trust and dissatisfaction.

---

### User Story 2 - User Personality LoRA Substrate (Priority: P1)

The runtime should maintain a separate fixed-size LoRA that accumulates only
from evicted user-message spans and represents the rhetorical tendencies of the
current user. This substrate is not versioned as a temporal stack and should be
continuously updated in place.

**Why this priority**: The system needs a durable but bounded substrate for
approximating user response style during counterfactual simulation without
mutating the base model or overloading the temporal memory stack.

**Independent Test**: Feed only user spans into the substrate and confirm the
runtime creates, updates, and exposes a dedicated user-personality adapter while
leaving temporal bucket semantics unchanged.

**Acceptance Scenarios**:

1. **Given** evicted spans containing user-authored messages,
   **When** the runtime ingests them for user-personality training,
   **Then** a dedicated user-personality LoRA receives Adam-backed updates and
   remains fixed-size and continuously modified.
2. **Given** evicted spans from system or tool output,
   **When** the runtime processes them, **Then** the user-personality LoRA is
   not updated from those spans.

---

### User Story 3 - DMN Counterfactual User Reply Simulation (Priority: P1)

DMN counterfactual evaluation should be able to draft a candidate message,
switch into a base-model-plus-user-personality serving stack, synthesize a
crude user reply, feed that reply back through counterfactual self-state
observation, and score whether the resulting shift moved the system closer to
the desired self-state.

**Why this priority**: This is the first feature that turns the user model into
an actionable counterfactual evaluator rather than a passive memory artifact.

**Independent Test**: Trigger a DMN counterfactual episode and verify that the
runtime produces a candidate outbound message, swaps to a user-simulation
serving stack with temporal LoRAs ablated, synthesizes a reply, applies that
reply on the counterfactual channel, and records the resulting trace.

**Acceptance Scenarios**:

1. **Given** a DMN counterfactual branch that evaluates a message variant,
   **When** user simulation is enabled, **Then** the system uses only the base
   model plus the user-personality LoRA for the simulated reply and excludes all
   model-specific temporal LoRAs from that pass.
2. **Given** a simulated user reply on the counterfactual channel,
   **When** the self-state evaluates the resulting change, **Then** the
   counterfactual trace records the simulated message, simulated reply, and the
   resulting self-state delta used for scoring.

## Edge Cases

- If the user-personality LoRA has little or no training data, simulation must
  degrade gracefully to a base-model-only or near-neutral-user pass while still
  recording low confidence.
- If DMN counterfactual simulation cannot decode a reply within bounded budget,
  the runtime must skip the simulation path rather than stall the loop.
- If the serving stack override fails, the runtime must restore the normal stack
  and continue without leaking the override into primary inference.
- If a user-model fragment represents hard-memory facts rather than preferences
  or rhetoric, it must not distort allostatic targets unless explicitly marked
  as allostatic.
- If a user message span is highly redundant, the user-personality LoRA update
  path must preserve existing redundancy suppression and bounded optimizer
  behavior.

## Requirements

### Functional Requirements

- **FR-001**: The system MUST evaluate the current user model surfaces in hard
  memory and self-state and extend them with higher-leverage typed user signals.
- **FR-002**: Hard memory `USER_MODEL` primitives MUST be able to encode
  user-preference, rhetorical-style, and interaction-policy residue in bounded
  typed summaries.
- **FR-003**: Self-state model summaries MUST expose bounded user preference and
  rhetorical estimates in typed structs and summary registers without replacing
  the existing social or user-outcome state.
- **FR-004**: The system MUST maintain a dedicated fixed-size user-personality
  LoRA substrate that is updated only from evicted spans sourced from user
  messages.
- **FR-005**: The user-personality LoRA MUST be continuously modified in place
  and MUST NOT be versioned through the temporal LoRA bucket stack.
- **FR-006**: The runtime MUST expose explicit serving-stack policy for
  temporarily running with base model plus user-personality LoRA while ablating
  all model-specific temporal LoRAs during simulated user replies.
- **FR-007**: DMN counterfactual evaluation MUST support a bounded simulation
  loop of candidate message generation, user-personality reply synthesis,
  counterfactual self-state observation, and scoring.
- **FR-008**: Simulated user replies MUST be treated as user events on the
  counterfactual channel and MUST NOT leak into primary-channel runtime state.
- **FR-009**: The user-personality simulation path MUST expose typed traces for
  candidate message text, simulated reply text, confidence, self-state delta,
  and whether temporal ablation was applied.
- **FR-010**: The planning/composition loop MUST be able to use the simulated
  reply outcome as part of counterfactual comparison without becoming a separate
  third loop.
- **FR-011**: The feature MUST remain bounded in adapter rank, update budget,
  simulation token budget, trace size, and failure fallback behavior.
- **FR-012**: The README, architecture docs, and developer docs MUST explain
  how richer user-model data is captured, how the user-personality substrate is
  trained, and how future tools can contribute user-model signals safely.

### Key Entities

- **User Model Fragment**: A hard-memory primitive summarizing stable user
  preferences, rhetorical tendencies, or interaction-policy observations.
- **User Preference Summary**: A bounded self-state summary of the current best
  estimates for user preferences, rhetorical tendencies, and uncertainty.
- **User Personality LoRA**: A fixed-size runtime LoRA continuously trained only
  from evicted user-authored spans and used only for DMN counterfactual reply
  simulation.
- **User Simulation Trace**: A typed record of candidate message generation,
  serving-stack override, simulated reply, confidence, and observed self-state
  delta.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Targeted tests can demonstrate that user-model hard-memory
  archives now capture more than social residue, including preference and
  rhetorical summaries.
- **SC-002**: Targeted tests can demonstrate that only user-authored evicted
  spans update the user-personality LoRA substrate.
- **SC-003**: Targeted tests can demonstrate that DMN counterfactual simulation
  runs with temporal LoRAs ablated and restores the normal serving stack after
  the simulated reply.
- **SC-004**: Targeted tests can demonstrate that simulated user replies
  produce counterfactual self-state traces without mutating primary-channel
  state.
- **SC-005**: Documentation clearly explains the new user-model surfaces,
  simulation path, and extension points for future tool-authored user-model
  signals.
