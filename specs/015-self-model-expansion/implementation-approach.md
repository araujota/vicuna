# Implementation Approach: Expanding Vicuña's Self-Model Without Losing Explicitness

## Core Recommendation

Expand the self-model in three coordinated ways:

1. Add a small number of new routing-critical scalar registers.
2. Add grouped typed profiles for the state that should not be expressed as a
   flat enum.
3. Add multi-timescale and forecast state so the runtime can represent progress,
   efficiency, and recovery instead of only local pressure.

This is a better architecture than simply multiplying scalar register count.

## Why "More Registers" Alone Is Not Enough

The user intuition is directionally correct: the system does need more internal
state. But turning every missing concept into a single new scalar creates three
problems:

- semantic drift: unrelated quantities end up with inconsistent update rules
- poor debuggability: it becomes hard to understand why a loop decision changed
- weak timescale handling: flat scalars do not naturally separate current state
  from trend and baseline

The right move is:

- more registers where fast routing needs them
- more structured profiles where semantic grouping matters

## Proposed Architecture

### 1. Preserve The Existing Fast Path

Keep the current register bank and prewrite/postwrite pipeline as the loop's
fast control layer.

Do not remove current signals such as:

- contradiction
- uncertainty
- goal relevance
- broadcast pressure
- broadcast inhibition
- follow-up continuation
- tool salience

Add only the new scalar summaries that will directly improve immediate routing:

- user satisfaction risk
- goal progress pressure
- answerability
- loop inefficiency
- recovery urgency
- preference uncertainty

### 2. Add Typed Self Profiles

Implement the new profile structs proposed in
[data-model.md](/Users/tyleraraujo/vicuna/specs/015-self-model-expansion/data-model.md)
inside `llama_self_state`.

The rule should be:

- if a concept is a loop-wide domain with multiple related factors, use a
  profile
- if a concept only needs one cheap routing scalar, use a register

This yields:

- compact loop control
- richer inspectable semantics
- cleaner public API and trace design

### 3. Add Horizon Slices

Each typed profile should keep:

- instant slice
- short EMA slice
- long baseline slice

Why:

- instant state explains the current event
- short state explains current trajectory
- long state explains persistent condition and prevents oscillatory overreaction

This is the minimum useful structure for efficiency-oriented control.

### 4. Add Forecast State

Introduce explicit bounded forecasts for:

- expected steps remaining
- expected inference cost remaining
- predicted change in user satisfaction
- predicted change in favorable-state divergence
- predicted goal-progress delta

These forecasts should be generated by explicit CPU-side rules at first. Later
evaluators can replace or calibrate them.

### 5. Add Prediction-Error Traces

Every forecast should eventually be compared with observed outcome.

This enables:

- later calibration
- later learned evaluator training
- debugging of poor self-estimates
- future self-improvement gating based on actual improvement rather than only
  intuition

## Recommended Update Flow

### Event Processing

1. Build the current prewrite or postwrite feature vector as today.
2. Update the existing register bank.
3. Derive instant typed profiles from current features, tool state, social
   state, working memory, and favorable-state context.
4. Update short and long horizon slices via explicit smoothing policy.
5. Derive scalar summary registers from the richer profiles where needed.
6. Emit forecasts and attach them to the current loop or event trace.
7. On later observations, compute prediction error against prior forecasts.

### Policy Placement

Keep the policy explicit in:

- `src/llama-self-state.cpp` for profile update rules
- `src/llama-cognitive-loop.cpp` for action use of the new surfaces
- public trace structs in `include/llama.h`

Do not bury motivation logic inside Active LoRA writes or opaque latent memory.

## How The New Profiles Help The Stated Goals

### User Satisfaction And User Goals

The new user-outcome and goal-progress profiles help separate:

- "the user is frustrated"
- "the user is confused"
- "the system is not making progress"
- "the task is progressing but trust is slipping"

That makes later user-aligned control and evaluation feasible.

### Returning To Desirable Self-State

The recovery profile makes self-repair more actionable by exposing:

- which favorable-state family is off target
- how fast recovery is currently happening
- how costly recovery is likely to be
- whether regulation debt is accumulating

This is stronger than a single aggregate favorable divergence.

### Becoming More Efficient

The efficiency profile is the key addition for the user's stated objective.

Without it, the runtime cannot explicitly represent:

- that it is taking too many iterations
- that it is repeating itself
- that a tool path is too costly
- that a clarifying question would reduce total loop cost

With it, later evaluators can optimize for fewer-step completion without having
to reconstruct efficiency from raw traces alone.

## Motivation For Self-Improvement

The self-improvement profile should not yet trigger autonomous retraining or
large policy shifts. It should first exist as bounded inspectable state.

Recommended fields:

- `update_worthiness`: is an internal change likely worth the cost right now?
- `expected_gain`: if changed, how much better should future behavior get?
- `evidence_deficit`: how weak is the current basis for intervening?
- `reversibility`: can this be safely undone or contained?
- `blast_radius_risk`: how likely is collateral degradation?
- `observability_deficit`: would we even be able to tell whether the change
  helped?

This lets governance later decide:

- defer
- repair
- gather information
- proceed with bounded internal adjustment

## Parity Requirements

### Prewrite / Postwrite

Preserve the current order. New profiles should be derived from the same event
and feature surfaces, not from a side system with different timing semantics.

### Favorable State

Preserve `llama_favorable_state_profile` as the public compact summary. Compute
it from richer recovery-aware internals rather than replacing it immediately.

### Counterfactual And Remediation

Preserve current trace concepts. The richer self-model should improve candidate
ranking and remediation selection, not force a redesign of those mechanisms.

### Active Loop And DMN

Keep the current action set and trace shape stable where possible. Use the new
profiles to improve scoring and tie-breaking.

## Incremental Rollout

### Stage 1: New Typed Profiles, No Loop Behavior Change

- add profile structs and horizon storage
- compute them in self-state
- expose them through read APIs and traces
- keep current loop decisions unchanged

### Stage 2: New Scalar Summaries

- derive a few new routing-critical registers from the profiles
- keep weights conservative
- validate no regressions in current decision tests

### Stage 3: Loop Integration

- feed efficiency, answerability, recovery urgency, and user-outcome summaries
  into Active Loop and DMN scoring
- keep the decision policy explicit and inspectable

### Stage 4: Prediction Error And Calibration Hooks

- record forecast versus observed outcome
- expose those traces for later evaluator work

## Testing Strategy

Implementation should add tests for:

- profile update correctness from known event sequences
- horizon slice behavior and smoothing
- forecast trace emission
- prediction-error recording
- favorable-state parity
- self-state import/export parity
- Active Loop parity with and without new profile integration

## Final Recommendation

The system should expand both the number and complexity of its registers, but
the design should be layered:

- a slightly richer scalar bank for fast routing
- typed profiles for semantically coherent domains
- horizon slices for trend and baseline
- forecasts and prediction errors for efficiency-oriented improvement

That is the strongest architecture for building a crude recursive
self-improvement substrate without giving up Vicuña's core advantage of
explicit, inspectable self-state.
