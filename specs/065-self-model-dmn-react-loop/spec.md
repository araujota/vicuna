# Feature Specification: Self-Model-Translated DMN ReAct Loop

**Feature Branch**: `065-self-model-dmn-react-loop (spec-only on current 060-service-user-migration worktree)`  
**Created**: 2026-03-20  
**Status**: Draft  
**Input**: User description: "we need to implement a seed change in how the DMN process works; it should essentially be a perpetual internal ReAct loop, functioning exactly the same as the active engagement one, but the 'prompt' should be generated by translating the state of the internal self-model into natural language. this is a crucial process, since the self-model can be expanded by the system itself, and we don't want to be 'dog fooding' the system by just saying 'generate a prompt from this self-model'. investigate methods for this with a robust research project. the full loop should be self state translation into NL -> this becomes the prompt for a non-user-facing loop with all the same functional lora/tool selection business -> every time the self-model is updated, the translation into NL occurs again and it becomes the prompt for a new phase of the DMN loop. the process must have the ability to reach out to the user via the telegram bridge as a first-class tool option with questions/comments/conclusions, etc, and this should not count as an active engagement response/stop the dmn loop. the overall goal here is the translation of mathematical representations of state to naturla language motivations; the distinction between a 'neurological' layer which underpins and mutally affects the 'conscious voice' generated by inference, but is not directly inspectable by it. do a robust research project and generate spec kit artifacts for this transition, including the cleanup of any code this will obsolete"

## User Scenarios & Testing

### User Story 1 - Hidden self-model becomes a bounded reportable motivation prompt (Priority: P1)

As a runtime maintainer, I need the DMN to receive a bounded natural-language
motivation prompt compiled from the typed self-model rather than a raw state
dump or an opaque "summarize yourself" generation, so the reportable DMN
workspace stays explicit, inspectable, and decoupled from the mathematical
state substrate that drives it.

**Why this priority**: This is the core architectural shift. Without an
explicit translation layer, the system is still dogfooding the hidden
self-model directly and cannot preserve the neurological/reportable
distinction.

**Independent Test**: Mutate typed self-state in a targeted way and verify the
runtime emits a deterministic `DmnPromptRevision` whose content changes only
through explicit translation policy, not through free-form summarization.

**Acceptance Scenarios**:

1. **Given** a self-model revision that changes contradiction, uncertainty,
   priorities, or discovered self-state, **When** the translation pipeline
   runs, **Then** it emits a bounded natural-language prompt built from explicit
   concept-selection and realization policy rather than directly exposing raw
   mathematical slots.
2. **Given** a DMN tick with no material self-model revision since the last
   prompt revision, **When** the scheduler evaluates whether to regenerate the
   DMN prompt, **Then** it reuses the current prompt revision instead of
   churning a new one.
3. **Given** self-model extensions introduce new typed fields or discovered
   dimensions, **When** those fields are admitted into translation input,
   **Then** only explicitly whitelisted concept mappings reach the prompt.

---

### User Story 2 - DMN reuses the active planner/tool runner as an internal ReAct loop (Priority: P1)

As a runtime operator, I need the DMN to run through the same planner-first
tool and functional-LoRA machinery used by active engagement, so background
cognition becomes a real internal ReAct loop instead of a separate
seed-and-winner-action heuristic.

**Why this priority**: The request is not for a prompt tweak; it is for a
structural convergence between active and background cognition.

**Independent Test**: Start a DMN phase from a translated prompt revision and
verify it drafts a plan, invokes tools, integrates results, and revises or
restarts on self-model updates through the same loop substrate as the active
path.

**Acceptance Scenarios**:

1. **Given** a translated DMN prompt revision, **When** the DMN enters a new
   episode, **Then** it uses the existing planner/tool plan-step substrate
   rather than selecting among `silent`, `internal_write`, `invoke_tool`, and
   `emit` candidates through dedicated DMN winner-action scoring.
2. **Given** a DMN episode integrates a tool result or internal write that
   changes self-state, **When** a new self-model revision is admitted, **Then**
   the translation layer emits a new prompt revision and the DMN starts a new
   phase from that prompt revision.
3. **Given** the active loop is idle while DMN work is ongoing, **When** the
   DMN continues through multiple plan steps, **Then** active-loop behavior and
   user-facing response accounting remain separate from DMN episode state.

---

### User Story 3 - DMN can reach the user through Telegram as a first-class tool (Priority: P2)

As a runtime operator, I need DMN-origin questions, comments, and conclusions
to flow through the Telegram bridge as an explicit tool action, so user contact
becomes an inspectable capability within the DMN ReAct loop rather than a
special background emit path that collapses loop state.

**Why this priority**: User outreach is part of the requested background loop,
but it must not be conflated with active engagement or terminate background
cognition.

**Independent Test**: A DMN episode selects the Telegram relay tool, delivers a
message through the bridge, records the delivery result as tool observation, and
continues background cognition without being counted as an active response.

**Acceptance Scenarios**:

1. **Given** a translated DMN prompt revision that surfaces a user-directed
   uncertainty or conclusion, **When** the planner selects the Telegram relay
   tool, **Then** the bridge receives a first-class tool request with explicit
   intent metadata and delivery is recorded as a tool result.
2. **Given** a DMN-origin Telegram message is delivered, **When** the runtime
   updates loop state, **Then** the delivery does not mark the active loop as
   answered and does not terminate the DMN loop.
3. **Given** Telegram delivery fails or is policy-blocked, **When** the DMN
   integrates the result, **Then** the failure remains a tool observation that
   the DMN may reason over in the next phase.

---

### User Story 4 - Obsolete DMN heuristic scaffolding is removed or demoted to explicit translation policy (Priority: P2)

As a maintainer, I need the old pressure-seed and DMN winner-action scaffolding
to be removed, narrowed, or repurposed into the new translation trigger policy,
so the runtime has one coherent background architecture instead of two
competing control schemes.

**Why this priority**: Leaving the current heuristic DMN stack in place would
reintroduce hidden policy conflicts and make the new design unreviewable.

**Independent Test**: Inspect runtime traces and codepaths after implementation
to verify DMN episode selection no longer depends on `assemble_seed`,
four-action DMN candidate scoring, or special-case background emit semantics.

**Acceptance Scenarios**:

1. **Given** the new DMN ReAct loop is active, **When** a maintainer inspects
   code and traces, **Then** prompt generation is attributed to a
   self-model-translation pipeline rather than `assemble_endogenous_seed`.
2. **Given** the DMN decides to contact the user, **When** the corresponding
   plan step is inspected, **Then** it appears as an explicit Telegram tool
   action rather than a dedicated background emit winner-action.
3. **Given** targeted native tests and docs are updated, **When** operators
   review the architecture, **Then** the obsolete heuristic surfaces and their
   replacements are called out explicitly.

### Edge Cases

- Self-model revisions arrive faster than a DMN episode can complete, requiring
  explicit supersession and restart semantics instead of prompt churn races.
- The translator receives newly discovered self-state dimensions whose concept
  mapping is not yet whitelisted.
- The translated prompt has no salient outward action but still needs to keep
  an internal episode alive or explicitly defer.
- DMN Telegram relay is available but anti-spam or social-state policy blocks
  delivery.
- Active engagement and DMN both want the same external tool family at the same
  time.
- The DMN self-model translation changes while a Telegram relay tool call is in
  flight.
- A self-model update is numerically real but below the configured materiality
  threshold for prompt regeneration.
- The runtime restarts while a DMN prompt revision and episode are both active.

## Requirements

### Functional Requirements

- **FR-001**: The runtime MUST define an explicit self-model-to-language
  translation pipeline that compiles typed self-state into a bounded natural
  language DMN prompt revision.
- **FR-002**: The translation pipeline MUST keep the hidden mathematical
  self-model non-directly inspectable by the DMN prompt consumer; only the
  translated prompt revision and typed translation trace may be exposed to the
  reportable loop.
- **FR-003**: The translation pipeline MUST derive its output from explicit
  CPU-side concept-selection, salience, ordering, and realization policy rather
  than by asking the model to summarize raw self-state.
- **FR-004**: The runtime MUST track self-model revisions and MUST regenerate
  the translated DMN prompt when a material self-model revision is admitted.
- **FR-005**: The runtime MUST support self-model revision materiality rules so
  insignificant numeric churn does not force unnecessary prompt regeneration.
- **FR-006**: The DMN MUST consume the translated prompt through the same
  planner/tool/functional-LoRA substrate used by the active loop, with a
  distinct DMN origin and output policy.
- **FR-007**: DMN episode planning MUST no longer depend on dedicated
  four-action DMN winner-action scoring as the primary behavior-selection path.
- **FR-008**: The runtime MUST support superseding an in-flight DMN episode
  when a newer translated prompt revision materially replaces its underlying
  self-model revision.
- **FR-009**: The DMN MUST expose Telegram relay as a first-class tool option
  with explicit request/result structs and policy gating.
- **FR-010**: DMN-origin Telegram relay MUST NOT count as an active engagement
  response and MUST NOT terminate the DMN loop by itself.
- **FR-011**: Telegram relay delivery or failure MUST be integrated as a normal
  tool observation available to later DMN phases.
- **FR-012**: The implementation MUST inventory and remove, demote, or
  repurpose obsolete DMN-specific heuristic surfaces, including endogenous seed
  assembly and special-case background emit action logic, wherever they would
  conflict with the new architecture.
- **FR-013**: The runtime MUST expose typed traces for self-model revisions,
  translation inputs, prompt revisions, DMN episode supersession, and Telegram
  relay tool activity so the new policy remains inspectable.
- **FR-014**: The implementation MUST update targeted native tests for self
  state, cognitive-loop planning, and server/tool integration to cover the new
  translation-triggered DMN lifecycle.
- **FR-015**: The implementation MUST update architecture and operator
  documentation to describe the neurological self-model layer, the reportable
  DMN prompt layer, Telegram relay behavior, and the removed heuristic DMN
  surfaces.

### Key Entities

- **Self Model Revision**: A typed revision marker describing the most recent
  material change to the hidden self-model and the domains that changed.
- **Self Model Translation Input**: The bounded, typed snapshot consumed by the
  translation compiler, including self-state summaries, deltas, and
  whitelisted extension concepts.
- **Reportable Concept Frame**: An intermediate representation of motivational
  or interpretive concepts extracted from the hidden self-model before natural
  language realization.
- **DMN Prompt Revision**: The natural-language prompt artifact compiled from
  reportable concept frames and consumed by the DMN planner/tool loop.
- **DMN Internal Episode**: One background ReAct episode bound to a specific
  prompt revision and planner/tool lifecycle.
- **DMN Telegram Relay Request**: A first-class tool payload for user-directed
  background outreach without collapsing the DMN loop into active engagement.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Targeted tests show that material self-model revisions produce
  deterministic prompt revisions with explicit translation traces, while
  immaterial churn does not.
- **SC-002**: Targeted tests show the DMN executes plan drafting, tool
  invocation, tool-result integration, and prompt-revision supersession through
  the shared planner/tool substrate.
- **SC-003**: Targeted tests and server traces show DMN-origin Telegram relay
  behaves as a tool result and does not increment active-response accounting or
  terminate background cognition.
- **SC-004**: Architecture docs and code inspection show the previous DMN
  endogenous seed and four-action winner-action stack no longer governs DMN
  behavior.
- **SC-005**: Operators can inspect the hidden-to-reportable translation path,
  prompt revision lineage, and DMN supersession decisions through explicit
  traces rather than opaque prompt text alone.
